      阿里月饼事件

      Post on 2016-09-19

      最近的阿里巴巴月饼事件在IT界闹得沸沸扬扬，事情大概是几个阿里的程序猿在公司内部抢购月饼活动中为了能够多抢购几盒月饼在程序中加了点代码，被公司发现，于是炸了，这几个员工两小时内光速被开除。

      其中一个匿名当事人是这么说的：“9月12日下午14时，听说开始秒杀月饼了，刚开始用手点发现打开就没了。看了下，有人用程序刷，于是我和旁边同事说，我也得写个脚本了，要不然抢不到了，随后写了一段JavaScript脚本，大概就是如果按钮变成了秒杀就狂点（和12306抢火车票插件类似），然后处理下验证码。写了个定时器抢16点的那一批。
      由于之前没进入过付款页面，以为和正常的秒杀页面一样，抢到了就会跳转到支付，所以完全没放在心上就去工作了。结果到16点一看傻眼了，那个页面秒杀可以一直点然后不跳转！一共是抢了16个（都没付款），赶紧给行政打电话说要退。后来的事情我也是猝不及防，16:30约谈，17:30解约合同就备好了，18:00走人，走之前还特意问了下需不需要交接工作，大家都不care，走的事情比较重要。”

      网络上对这件事件众说纷纭，但矛头都是指向阿里，有的说：“说轻点儿，是机制僵化，说重点儿，这是不拿码农当人”，还有的说：“阿里的政治氛围有点过了，阶级斗争这根弦绷得太近，什么问题都能上升到价值观问题”。几乎所有的人都在说阿里的不是，单是真的就是和这些人说的那样吗，当然不全是，个人觉得，下面就来看一下简单的demo从程序的角度来分析这件事情。

      < button class=”no” >活动还未开始< /button >

      setTimeout(function(){
    var x=document.getElementsByTagName('button')[0];
    x.innerHTML="可以开抢了";
    x.setAttribute("class","yes");
    x.onclick=function(){
        console.log("恭喜抢到了已一盒月饼！");
    }
},10000);

      //这段放在控制台执行
      var h=100;
      var t=setInterval(function(){
      var x=document.getElementsByTagName(‘button’)[0];
      if(x.getAttribute(“class”)==”yes”){
      clearInterval(t);
      for(var i=0;i < h;i++){
      x.onclick()
      }
      alert(“你抢了”+h+”盒月饼，溜得飞起！”)
      }
      },1);

      前面一段代码模拟的是抢购活动，意思是在十秒之后开始抢购月饼，后面的代码是计时器，每0.001s触发一次，使用它来监听是否处于抢购状态，如果是的或触发抢购时间方法。由此可见，只需要在控制台写上一段代码然后执行就可以实现抢购，但事实并非如此，如果只是这么简单就能抢到的话那抢购就没有什么意义了，在抢购中通常都会需要在点击抢购后需要在输入验证码，这样是为了防止程序不断抢购的现象。

      但是在阿里月饼事件中，这几个程序猿并不是单纯的些一些js代码抢购月饼的，上面事件当事人是这么说的大概就是如果按钮变成了秒杀就狂点（和12306抢火车票插件类似），然后处理下验证码。。如果只是由于写一段js代码抢到月饼的话，那只能说这是程序的BUG，这几个程序猿好像并没有这么大责任，但是当事人已经明确的说了处理过验证码，这里的处理可能是传验证码值过来然后js填入，也可能是填入特定的验证码，验证是放行，但不管是哪种方式都必须更改后台代码，当事人是利用职务的便捷给自己谋取利益，从这个角度来看这几位当事人确实应该接受惩罚，但是因为这个原因而对这几个员工执行开除处理是否合理，这里就不便多说，毕竟是别人家公司，而且每个都有每个人的看法，不能以自己的价值观去批判他人。

      个人看法，不喜勿喷。
